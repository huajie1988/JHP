实现 PhpParserBaseVisitor 的子类

一、补齐语言核心特性
抽象类与接口的完整支持

当前 classDeclaration 已经能识别 Interface，但在方法生成里尚缺少对接口的完整处理（例如接口中方法默认 public abstract，常量自动 public static final）。

抽象类需要检查 modifier 是否为 abstract，输出 abstract class 并允许抽象方法。

接口允许多继承（extends 多个接口），当前实现只有单继承，需要扩展。

实现接口的类需强制实现所有抽象方法，可在 Visitor 中简单的静态校验。

match 表达式

语法文件已包含 matchExpr，与 PHP 8 的 match 对应，功能类似 switch 但可返回值。

翻译为 Java 的 switch 表达式（Java 14+）或 if-else 链，并处理默认分支和逗号分隔的多条件。

字符串内嵌变量

复杂的双引号字符串和 heredoc 中嵌入 $var 或 {$expr} 尚未完全支持，目前只能简单输出。

应实现为字符串拼接和 JhpRuntime 格式化方法的调用。

instanceof 表达式与类型检查

语法已有 InstanceOf，但尚未在 ExpressionProcessor 中生成 instanceof 代码，需添加分发处理。

Throw 表达式

语法文件中 Throw expression 被注释掉了，如果需要支持可在表达式内允许 throw，生成 Java 的 throw（或使用 JhpRuntime.throwException）。

二、增强类型系统与推断
更智能的类型推断

目前对内置函数的返回类型需要手动声明（BUILTIN_RETURN_TYPES），可以考虑读取配置文件或外部定义。

对泛型容器（如 List<: User :>、Map<: string, List<: int :> :>）的元素推断已经较好，但对复杂嵌套仍需更多单元测试。

对联合类型 int|string 可降级为 Object 并给出警告，或支持可选的联合类型转换。

严格的可空类型 (?int)

检查参数或返回值是否为 nullable，翻译为 Java 包装类型还是基本类型？目前统一使用包装类，应能正确处理 null。

三、提升翻译质量与鲁棒性
更丰富的内置运行时函数库

持续扩展 JhpRuntime 中的方法，覆盖常用 PHP 函数如 array_merge, sort, implode, explode, preg_match 等，并在 RUNTIME_FUNCTIONS 中注册。

为这些函数提供自动类型推断（通过 BUILTIN_RETURN_TYPES）。

错误处理与恢复

当前编译器遇到致命错误会抛出 JhpCompilationException 并中止。可以增加非致命错误的收集机制（如警告列表），并在最后输出，类似 javac 的行为。

对语义错误（如类型不匹配、未定义变量）给出更精确的行号和信息。

代码优化：

对生成的 Java 代码进行简单的后处理（如删除不必要的强制转换、减少冗余括号）。

当类型明确时优先使用基本类型（如 int, double）而不是包装类，以提升性能。但鉴于目前基本类型映射为包装类（Integer, Double）有利于 null 安全和泛型，可保持现状，或作为可配置选项。

四、工具链与集成支持
更灵活的编译模式与项目管理

当前的模式 1~3 仅支持单文件或单类。可以扩展模式，支持编译整个项目（多 PHP 文件到一个 Java 项目），自动处理命名空间到目录结构的映射。

添加模式 4：将 PHP 文件编译为独立的 Java 类，但不强制生成 main 方法，而是作为库。

集成构建工具

提供 Maven 插件或 Gradle 插件，方便将 PHP 源文件作为 Java 子模块编译。

输出 jhp-runtime.jar 作为稳定依赖，并自动管理版本。

标准库映射

将常见的 PHP 类库映射到 Java 标准库（如 PDO → JDBC），可通过注解或配置文件实现，减少对 JhpRuntime 的依赖。

五、文档与测试
完整的测试用例覆盖

编写大量 PHP 测试文件，涵盖所有已实现特性，确保翻译正确性。

自动化测试框架：编译 PHP 文件，运行生成的 Java 类，验证输出。

用户文档

编写清晰的使用指南、语法差异说明、限制列表、配置方法。

提供示例项目，展示如何在 Spring 中使用编译生成的类。

优先级建议
立即开始：抽象类/接口完善、match 表达式、字符串内嵌变量、修复当前已知的 Bug（如刚刚解决的注解导致属性丢失问题）。

短期：丰富运行时函数、增强类型推断、改进错误报告、添加单元测试。

中期：多文件项目编译、工具链集成、标准库映射。

这样可以让编译器更接近实际生产使用，并为后续生成 JVM 字节码的后端奠定坚实基础。



我们已完成PHP转Java编译器的大部分核心功能：变量、表达式、控制流、函数、类成员、静态成员、异常、泛型、命名空间、导入、注解等。接下来可以按以下路线继续开发：

优先级1：完善运行时库（JhpRuntime）

补全常用的PHP内置函数（如字符串、数组、数学、日期等），通过内置函数映射表翻译为JhpRuntime.xxx调用。

设计运行时库的jar打包和分发机制，使其能独立于编译器项目使用。

优先级2：OOP增强

抽象类和抽象方法的翻译。

接口（interface）的翻译，包括接口的默认方法。

访问修饰符的细化（protected、private在继承中的可见性检查）。

多态和方法调用的动态绑定处理（目前静态方法调用已支持，实例方法调用已支持，但可能需要更准确的类型推导以保证重写正确）。

优先级3：错误与诊断

增强类型不匹配、未声明变量、静态上下文错误等的报错，目前部分错误通过System.err输出，建议统一为JhpCompilationException并停止编译。

添加源码行号/列号信息到错误消息中（需结合ANTLR的Token信息）。

优先级4：更多语法支持

match表达式（PHP 8的match，可翻译为Java的switch语句或表达式）。

字符串内嵌变量（双引号字符串中的$var），目前可能未处理或只处理简单情况，需转换为Java字符串拼接或String.format。

instanceof运算符（已支持解析，但翻译需增加类型转换和instanceof关键字）。

clone表达式（可翻译为Java的clone()方法调用）。

枚举（enum）的翻译（PHP 8.1枚举，可翻译为Java枚举类）。

优先级5：工具链与集成

提供命令行脚本或Maven/Gradle插件，方便集成到开发流程。

生成代码时可配置包名、导入、是否生成main方法等。

支持多个PHP文件编译为一个Java项目（多文件模式，结合命名空间和自动加载）。

优先级6：优化与性能

对数组下标访问根据类型生成更高效的.get()调用而非arrayGet（已部分实现，可继续完善）。

常量折叠、简单表达式预计算。

减少不必要的Object类型推断，利用符号表和强制转换提示更精确的类型。

下一步建议从优先级1开始，完善运行时库，因为这是生成代码可运行的基础。之后逐步完善OOP和错误诊断，让编译器更健壮。
